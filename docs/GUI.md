# Видение интерфейса PVTcalc (черновик, обсуждение)

Это **не финальная спецификация**, а зафиксированный итог обсуждения концепции интерфейса (2026-07-15). Цель — не потерять договорённости и открытые вопросы к следующей сессии. Дополняется по мере обсуждения. План работ по движку — в [`BACKLOG.md`](BACKLOG.md); здесь — про UI.

## Что уже зафиксировано (не переобсуждаем)

- **Главный пользователь** — автор; MVP «для себя», но с прицелом показать коллегам. Пока — **localhost** (деплой для доступа коллег по ссылке — вопрос на потом, не закладываем многопользовательность сейчас).
- **Стек** — ✅ **решено 2026-07-16: DearPyGui** (нативный Python-GUI). В запасе — **PySide**, крайний вариант при необходимости деплоя «по ссылке» — пересадка на веб (FastAPI+React/Plotly). Ключевое следствие запасных вариантов: UI строится **поверх фреймворк-независимого слоя** (`AppState`/ViewModel над `calc_core`), View на DearPyGui — тонкий; смена стека переписывает только View. В любом варианте — поверх готового сервисного слоя `calc_core` (`CompositionalModel` + фасады `phase_envelope`/`experiments` + `flash`).
- **Хранилище**: `models.json` (можно всё в одном файле). Нужны метаданные модели (месторождение, скважина, дата отбора) — не приоритет для v1.

## Два главных драйвера дизайна

### 1. Реактивный вычислительный граф
- **Корневой узел** = состав + свойства (полностью редактируемый — см. «Редактирование состава»).
- От корня — узлы **экспериментов** (флэш, CCE, DLE, сепарация, огибающая, критическая точка, Psat), рядом с каждым — **сравнение с лабораторными данными**.
- **Реактивность**: правка корня → зависимые узлы помечаются «устаревшими». Стратегия пересчёта — **ленивая инвалидация** (пометить устаревшим) + кнопка **«пересчитать всё»**, а не авто-пересчёт всего на каждый чих (огибающая считается десятки секунд). Обязательны **индикатор прогресса и отмена**.
- Граф хорошо ложится на сервисный слой: каждый метод фасада (`flash`/`cce`/`dle`/`ssm`/…) = вычисление одного узла (вход = параметры, выход = результат, уже логируется в `result_store_object`).

### 2. Полностью настраиваемое рабочее пространство
- Главная претензия к PVTSim: **нельзя настроить под себя** — много лишних кнопок/полей/графиков, нет гибкости («как из окошка подводной лодки»).
- Хочется: панели/вкладки, которые можно **открыть, схлопнуть, переставить**. Вкладку модели открыл, когда нужна; не нужна — свернул. В рабочей области всё **динамично и настраиваемо**.
- **Раскладка панелей — часть состояния сессии**: при возврате к модели восстанавливается не только числовой результат, но и раскладка/открытые графики (продолжить с того же места).

## Стиль и выбор фреймворка (✅ решено 2026-07-16)

Выбран **DearPyGui** (нативный Python immediate-mode GUI поверх Dear ImGui + implot/imnodes). В запасе — **PySide**; крайний вариант, если понадобится доступ коллегам «по ссылке», — пересадка на **веб** (FastAPI + React/Plotly). Обоснование: встроенный node-editor (под вычислительный граф) + докинг панелей (под настраиваемое пространство), чистый Python поверх `calc_core` без API/JS, эстетика нравится автору. Известный минус — десктоп, не браузер; для localhost-v1 некритично.

**Тема оформления:** глобальная **светлая** тема (`gui/view/theme.py::build_light_theme`, палитра по мотивам ImGui `StyleColorsLight`, привязывается через `dpg.bind_theme` в `run()`) — по предпочтению автора.

**Ключевое следствие запасных вариантов — архитектура в два слоя:**
- **Фреймворк-независимый слой** (`gui/app_state.py` + `gui/services/`) знает только `calc_core`: состояние графа, статусы узлов, репозиторий моделей, запуск расчётов. **Не импортирует DearPyGui.**
- **Тонкий View** (`gui/view/`) — единственное место, знающее про DPG: рисует состояние, шлёт команды. Смена стека (PySide/веб) переписывает только `gui/view/`.

Оба запасных стека ложатся на тот же слой состояния/сервисов.

## План работ v1 (фазы)

Пакет GUI — top-level `gui/` (рядом с `calc_core/`, движок не трогаем). Запуск: `python -m gui` (зависимость `dearpygui`, опциональная группа `[gui]` в `pyproject.toml`).

- **Фаза 0 — Каркас** *(готово, ветка `gui/phase-0-1-dearpygui`)*: пакет `gui/`, точка входа `python -m gui`, главное окно с **закреплённой раскладкой** (слева прибитая панель дерева, справа рабочая область, снизу статус — без плавающего докинга), `AppState` (граф Проект→Модель→Вариант→Узел + статусы), файл сессии `gui_session.json`. Весь видимый текст — на английском.
- **Фаза 1 — Дерево + загрузка моделей** *(готово)*: левое дерево, список моделей из `models.json` через `Composition.from_db` (репозиторий `gui/services/model_repository.py`), открытие модели → таблица состава (корневой узел «Состав»).
- **Фаза 2 — Корневой узел (состав)** *(готово)*: рабочая область разбита на **вкладки** (tab-bar):
  - **Properties & correlations**: выбор EOS (`EOSType`) и корреляций C7+ (`gui/services/composition_service.py` — списки методов + обёртки), редактируемая таблица состава (inline-поля zi и свойств, правка через `edit_component_properties`), нормировка zi, кнопка «Recalculate properties» → `evaluate_composition_data`.
  - **BIP matrix**: полная симметричная матрица NxN, каждая ячейка — редактируемое поле; правка (i,j) зеркалится в (j,i) через `edit_bip_for_components`, диагональ фиксирована 0. Рендер матрицы 40×40 ~15 мс.
  - Смена корреляции помечает узел `STALE`, пересчёт → `FRESH`. Immediate-правки (zi/свойство/BIP) не дёргают перерисовку (без потери фокуса).
  - **Обе вкладки живут в отдельном окне** «Composition — <модель>», открываемом двойным кликом по модели в дереве (см. «Навигация» в Информационной архитектуре). Центральная рабочая область — под эксперименты.
- **Фаза 3 — Узел флэша** *(готово)*: в центральной области — ввод P (бар)/T (°C), кнопка «Run flash». Расчёт через `gui/services/flash_service.py` → `VLE.Flash` (обрабатывает и одно-, и двухфазный случай) в **воркер-потоке**; UI не висит, показывается `loading_indicator` + «Cancel». Результат применяется в главном потоке через `dpg.set_frame_callback`-опрос (DPG-safe). Панель результата из `FlashResult`: таблица Vapor/Liquid (доля фазы, M, молярный объём/плотность, density, Z, вязкость). **Реактивность (задел Фазы 4)**: правка состава/EOS/пересчёт помечают посчитанный флэш `STALE`. Отмена: движок непрерываемый, поэтому результат отбрасывается по завершении потока (UI остаётся отзывчивым). Узел `flash` заводится при открытии модели, `upstream=["composition"]`.
- **Фаза 4 — Реактивность + схема-граф**: ленивая инвалидация (правка корня → узлы `STALE`) + «пересчитать всё»; небольшая схема-граф модели (imnodes) — визуальная часть гибрида.
- **Фаза 5 — Сохранение сессии**: раскладка панелей + открытые узлы + последний результат в `gui_session.json`; восстановление при возврате.

**Заметки по движку под UI (уточнено 2026-07-16):**
- Чтение `models.json` для UI идёт через существующий `Composition.from_db` (`list_models()` + доступ по ключу). Пустая заготовка `Utils/Import.py::DBModelImport` для этого **не нужна** — не блокер MVP.
- Для узла флэша выбрать `VLE.Flash` (однофазная ветка есть), а не `CompositionalModel.flash` (ждёт двухфазную точку, T в K).

## Информационная архитектура

- **Слева — дерево**: `Проект → Модель → Варианты`.
  - **Вариант** = форк модели с другими свойствами / корреляциями / EOS / (в будущем) лампингом.
- **Навигация (реализовано 2026-07-16)**: в дереве **одиночный клик** по модели — выбор (в центральной рабочей области показываются её эксперименты); **двойной клик** — открывает **отдельное окно** «Composition — <модель>» (перемещаемое/закрываемое) с вкладками Properties/BIP. Центральная область отдана под флэши/эксперименты, редактор состава живёт в своём окне.
- **Рабочая область** — гибрид: дерево слева + центральная область экспериментов + отдельное окно редактора состава. Схема-граф модели (визуальная часть гибрида) — Фаза 4.
- **Возврат к модели** → результаты предыдущей сессии + раскладка панелей.

## Редактирование состава (корневой узел)

- **Ввод состава — разными способами**: E300 (Eclipse 300), Excel (есть `calc_core/Utils/CompositionLoader.py::CompositionExcelLoader`), ручной ввод, загрузка сохранённой модели.
- **Выбор корреляций свойств C7+** (Tc/Pc/ω/Vc/shift/Kw — pedersen/kesler lee/riazi и т.д.) — **должен быть в UI**.
- **Выбор EOS** — **должен быть в UI**. В движке — PREOS/SRKEOS/BRSEOS через сеттеры в одном классе (уравнение можно свести к одной из трёх реализаций); живой сейчас — Брусиловский.
- **Ручное редактирование свойств компонент и BIP** — есть в движке (`Composition.edit_component_properties`, `edit_bip_for_components`); нужно в UI (это часть будущего тюнинга под факт).

## Расчёты (приоритет для MVP)

- **Начать с флэша** (одна точка). Остальные эксперименты — дальше.
- Ввод ступеней — **вручную** или **списком/таблицей**.
- **Прогресс + отмена — обязательны.**

## Визуализация

- Обязательны для начала: **P-T фазовая огибающая** (с критической точкой), **CCE** (относительный объём / сжимаемость от P), **DLE** (Bo и Rs от P).
- Про **шаблоны графиков** (отраслевой/PVTSim-вид) — договорились обсудить отдельно.
- (Не зафиксировано, к обсуждению: наложение нескольких прогонов на один график, интерактив.)

## Фактические данные и сравнение

- Источник: **лаборатория** или **данные по отбору пробы**, разные форматы.
- **Граница MVP**: **наложение факта на расчётные кривые + таблица отклонений**. Достаточно для v1.
- Регрессия/автоподбор EOS — **не в MVP** (отдельная крупная фича, см. ниже).
