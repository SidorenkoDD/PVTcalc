# Видение интерфейса PVTcalc (черновик, обсуждение)

Это **не финальная спецификация**, а зафиксированный итог обсуждения концепции интерфейса (2026-07-15). Цель — не потерять договорённости и открытые вопросы к следующей сессии. Дополняется по мере обсуждения. План работ по движку — в [`BACKLOG.md`](BACKLOG.md); здесь — про UI.

## СТАТУС-16 на 2026-07-19: навигация по дереву моделей

Корневой узел активной модели теперь отмечается ASCII-маркером `[A]`, а
контекстное меню корня содержит **Save model**. Сохранение адресуется по
идентификатору выбранного узла, поэтому из дерева можно сохранить неактивную
модель и не перепутать варианты. Пункт отключён для незагруженных или уже
сохранённых моделей.

Клик по категории расчётов внутри раскрытой неактивной модели сначала делает
эту модель активной, а затем раскрывает категорию; статусная строка подтверждает
переключение. Перед удалением проекта дополнительно показывается предупреждение
о количестве загруженных моделей с несохранёнными изменениями.

## СТАТУС-15 на 2026-07-19: безопасная вставка столбца замеров

Пустой или недоступный системный буфер теперь показывает сообщение в строке
статуса и не прерывает приложение. Если перед вставкой сфокусировать ячейку
Lab Data, значения из скопированного столбца записываются вертикально начиная
с этой ячейки; недостающие строки создаются автоматически, а соседние колонки
существующих точек сохраняются. Помимо TSV, обработан плоский список вида
`100, 90, 80`, который иногда возвращают clipboard-мосты.

Если ячейка не выбрана явно, вставка начинается с первой строки и первой
колонки уже созданной таблицы. Клавиши Left/Right/Up/Down переключают фокус
между соседними ячейками Lab Data. Ctrl+V непосредственно в ячейке также
разворачивает многострочный/сплющенный столбец в отдельные точки.
Добавление, удаление, очистка и редактирование строк входят в общий undo/redo
стек активной модели и сохраняются в workspace-сессии.

## СТАТУС-13 на 2026-07-19: проекты и независимые варианты моделей

На стартовом экране Projects теперь отображаются именно **проекты**, а не отдельные записи моделей. Одна строка проекта агрегирует число моделей, поля, EOS и сводку расчётов. Для старого `models.json` без метаданных `project_id` каждая модель автоматически рассматривается как отдельный проект, поэтому миграция не ломает существующие файлы.

После открытия проекта в левом дереве показываются все его модели соседними корневыми узлами. На корневом узле модели доступен пункт **Duplicate model...**. Копия получает уникальные `Model name` и `Model id` (например, `Original (copy)`, затем `Original (copy 2)`), но их можно изменить вручную. Проверка имени выполняется без учёта регистра и лишних пробелов, а `model_id` должен оставаться валидным и не совпадать с существующим.

Копирование — атомарная read-modify-write операция через `ModelStore`, поэтому параллельные изменения не затираются. Переносятся состав, EOS, `T_res`, поле, корреляции и идентификатор проекта; история расчётов и GUI-workspace намеренно очищаются. Если исходная модель имеет несохранённые правки, UI предлагает **Save source & duplicate** и сохраняет их перед форком. Контекстное меню Projects удаляет проект целиком с подтверждением и резервной копией.

## СТАТУС-12 на 2026-07-19: фактические данные и компактная сетка графиков

Для каждого узла `CCE / DLE / Separator` добавлен сворачиваемый блок **Lab Data (measured)**. Это намеренно не отдельная глобальная вкладка: замеры принадлежат конкретному типу и прогону, поэтому их нельзя случайно перепутать между экспериментами. Набор колонок подбирается по типу эксперимента, строки добавляются/удаляются/очищаются, пустые ячейки допускаются, а значения валидируются как конечные числа. Данные сохраняются в `params["lab_data"]` через существующий workspace/session codec и не загрязняют состав модели или `node.result`.

На вкладке **Chart** кривые теперь раскладываются карточками по две в строке. Фактические точки накладываются на расчётную линию отдельной scatter-серией с суффиксом `(lab)`; диапазон давления учитывает обе серии. Добавление дополнительных кривых и изменение замеров сразу перестраивает только сетку графиков, без пересборки всего workspace.

Добавлены headless-проверки хранения замеров и рендера Lab Data/двухколоночной сетки. Нерегрессионный набор: **144 теста зелёные**, `ruff` и `mypy gui` проходят.

## СТАТУС-11 на 2026-07-19: безопасное восстановление GUI-сессии

`gui/session.py` нормализует входной JSON перед созданием `SessionState`: неверный корень, идентификатор модели, размеры окна, workspaces и legacy-коллекции не могут уронить запуск или передать DPG некорректные размеры. Повреждённая сессия откатывается к defaults с предупреждением в логах.

`gui/workspace_codec.py` теперь пропускает отдельные мусорные записи узлов, неизвестные типы, неверные `params/upstream/open_tabs`, отрицательные/нечисловые счётчики и испорченные legacy-списки. Валидные соседние узлы восстанавливаются; `RUNNING` по-прежнему переводится в `STALE`. Headless-тест открывает Projects → workspace с таким файлом и проверяет отсутствие исключения. Полный набор: **145 тестов зелёные**.

## СТАТУС-10 на 2026-07-19: гранулярная перерисовка workspace

`AppState` теперь публикует типизированные `StateChange` с областью изменения (`NAVIGATION`, `MODEL_LIST`, `WORKSPACE`, `NODE`), сохраняя прежний callback API для сервисных слушателей. Composition root подписывается на эти события и не вызывает полную пересборку экрана там, где она не нужна.

Для событий конкретного узла DearPyGui обновляет дерево и тело только соответствующей открытой вкладки. Ее tab-bar, соседние вкладки, поля ввода и графики сохраняют свои контейнеры; заголовок вкладки и breadcrumb синхронизируются локально. Поэтому переходы `RUNNING → FRESH/STALE` фоновых расчётов не пересоздают весь workspace и не затрагивают локальное состояние соседних вкладок; тело целевой вкладки перестраивается осознанно, чтобы показать новый статус/результат. При открытии модели восстановление сессии выполняется до единственного события навигации, что убирает двойной рендер.

Добавлены headless regression-тесты на область событий, сохранение контейнеров вкладок и существующий DPG layout. Полный набор: **141 тест зелёный**, затронутые GUI-модули проходят `ruff` и `mypy gui`.

При повреждённой базе Projects также даёт явную кнопку **Refresh models**: после исправления `models.json` список можно восстановить без перезапуска приложения.

## СТАТУС-9 на 2026-07-18: единый безопасный ModelStore

Все рабочие пути чтения и записи `models.json` сведены в `calc_core/Utils/ModelStore.py`. Хранилище проверяет корень и записи моделей, поддерживает `schema_version=1` с миграцией legacy-записей в памяти, отвергает повреждённый JSON с координатами ошибки и не допускает `NaN`/`Infinity` или несериализуемые значения. `ModelJSONDB`, `Composition.from_db`, Projects и сохранение отредактированного состава используют один контракт вместо собственных вариантов `json.load`/guard/backup.

Запись выполняется атомарно под межпроцессным lock; перед ней ротируются три резервные копии (`.bak`, `.bak.1`, `.bak.2`). Долгоживущий `ModelJSONDB` проверяет revision и не перезаписывает более свежую версию файла. Read-modify-write операции редактирования и удаления держат единый lock на всю транзакцию. Формат корня не изменён (`model_id -> record`), а версия добавляется в запись только при следующем реальном сохранении — простое открытие не переписывает пользовательский файл.

Повреждение базы больше не обрывает запуск GUI: `AppState` публикует диагностическое состояние, а Projects показывает предупреждение и остаётся доступным, не пытаясь автоматически затереть или «исправить» данные. Покрыты legacy migration, schema errors, координаты битого JSON, ротация backup, lock ownership, конкурентное сохранение, восстановление состояния и реальный рендер баннера в headless DearPyGui. Полный набор: **139 тестов зелёные**; затронутые модули проходят `ruff`, `mypy gui` — без ошибок.

## СТАТУС-8 на 2026-07-18: явный ViewContext и честные Settings

Feature-view больше не объявляют скрытые зависимости `_state/_jobs/_session/callback: Any`. Добавлен `gui/view/contracts.py`: immutable `ViewContext(state, session, jobs)`, runtime-checkable `ViewHost` и типизированная база `ContextBoundView`. `PVTcalcApp` создаёт контекст один раз; общие методы feature-view проверяются `mypy`, а дальнейший переход от mixins к объектной композиции можно делать по одному экрану.

Settings приведён в соответствие фактическому поведению движка: окно переименовано в **Engine constants (read only)**, показывает только значения `engine_defaults()`, все поля readonly, Save/Reset и неиспользуемое сохранение `gui_settings.json` удалены. Редактирование вернётся только вместе с явным `EngineConfig`, реально передаваемым солверам.

Headless smoke проверяет `ViewHost`/`ViewContext`, сборку основных вкладок и readonly-конфигурацию каждого поля Settings. После удаления трёх тестов неиспользуемого settings persistence и добавления актуальных проверок: **125 тестов зелёные**, `ruff gui tests` и `mypy gui` проходят.

## СТАТУС-6 на 2026-07-18: стабилизация, сохранение и разделение слоёв

**Актуальная точка останова.** Ветка `gui/phase-0-1-dearpygui`, **127 тестов зелёные**, изменения пока не смёржены. На этом проходе новая предметная функциональность намеренно не добавлялась: исправлены ошибки текущего UI, усилена сохранность данных и вынесена инфраструктура из DearPyGui-view.

Сделано:

- Редактирование состава, EOS, BIP и корреляций теперь помечает модель как изменённую и действительно сохраняется в `models.json`: есть **File → Save model**, `Ctrl+S`, отложенное автосохранение и сохранение при штатном выходе. Грязные модели отмечены `*` в Projects и дереве.
- Все JSON-записи модели, сессии и настроек атомарны (`tempfile` в той же директории + `fsync` + `os.replace`); перед изменением базы сохраняется `models.json.bak`. При повторном сохранении сохраняются `created_at`, метаданные и результаты, добавляется `updated_at`. Выбранные корреляции записываются уже при создании/импорте модели.
- Фоновые расчёты адресуются стабильным `NodeRef(model_id, variant_id, node_id)`: переключение модели больше не может направить результат в чужой узел. Диспетчер потока вынесен в `gui/calculation_coordinator.py` и тестируется без DPG.
- Сессия обновлена до **v3**. Общий codec в `gui/workspace_codec.py` сохраняет и восстанавливает все типы узлов, точные id, результаты, статусы, параметры, счётчики, открытые вкладки и Compare; старые v1/v2 читаются. Незавершённый `RUNNING` после перезапуска становится `STALE`. Сессия сохраняется с debounce, а не только при выходе.
- Undo/Redo теперь сохраняет счётчики экспериментов/огибающих и не делает тяжёлую глубокую копию уже готовых результатов. Прямые мутации View заменены командами `AppState`.
- Excel без заголовка уважает выбранный лист. E300 сначала показывает отчёт (нераспознанные компоненты, пропущенные доли, предупреждения) и требует явного подтверждения потерь. Ввод P/T/долей/сеток/ступеней проверяется на `NaN`, `inf`, знак, диапазон и размер до запуска движка.
- Однофазный Flash сохраняет совместимые с экспериментами доли фаз, но дополнительно возвращает `phase_type = vapor|liquid|ambiguous`; UI больше не подписывает любую стабильную фазу как Liquid.
- Исправлены согласованные ошибки ядра: `273.15`, стандартная температура `293.15 K`, точная `R`, бинарная запись `ResultStore`, `LengthMismatchError` с legacy-алиасом и возвращаемое значение `PhaseIdentificator`. Численные regression-baseline обновлены осознанно.
- Добавлены framework-independent тесты координатора/workspace codec/атомарного сохранения и headless smoke реального DearPyGui context/layout/menu. CI запускает весь набор под Python 3.11/3.12/3.13 через Xvfb. Dev-набор включает `pytest-cov`, `ruff`, `mypy` и DearPyGui.

Текущее разделение GUI:

- `gui/app_state.py` — состояние и команды, dirty/history/stable node addressing;
- `gui/calculation_coordinator.py` — жизненный цикл одной фоновой задачи;
- `gui/workspace_codec.py` + `gui/session.py` — независимая от DPG сериализация и атомарная сессия;
- `gui/services/` — валидация и адаптеры над `calc_core`;
- `gui/view/app.py` — DPG composition root; панели и модальные окна вынесены в отдельные `*_view.py` (см. СТАТУС-7).

Известные границы этого этапа: отмена численного расчёта пока означает «не принимать завершившийся результат», потому что сами солверы не имеют cancellation token. Constants показываются только для чтения; редактирование требует отдельного `EngineConfig`, реально передаваемого солверам. `RootChooser.py` и незавершённый CVD остаются намеренно изолированными согласно `CLAUDE.md` и не включены в GUI-путь.

## СТАТУС-5 на 2026-07-17: E300 экспорт/импорт, Settings, фикс импорта Excel

**Актуальная точка останова.** Ветка `gui/phase-0-1-dearpygui`, **114 тестов зелёные** (+10), НЕ смёржено в master.

Сделано (четыре пункта по предложениям автора):
- **Фикс бага импорта Excel** (коммит `d71c3b5`): после «Load as composition» форма нового флюида не открывалась. Причина — модалка-поверх-модалки: новая `modal`-модалка, созданная в том же кадре, где закрыли предыдущую (Excel-предпросмотр), у ImGui остаётся скрытой (`show=False`, OpenPopup не срабатывает). Фикс — открывать форму на следующем кадре через `dpg.set_frame_callback`. Тот же паттерн стоит помнить для любых переходов «закрыл модалку → открыл другую».
- **Экспорт E300** (меню сверху **Export → Export model…**): новый `calc_core/Utils/E300Export.py::E300Exporter` (портирован из авторского модуля на старом `calculations.*`; адаптирован под текущий API — `Composition.composition`/`composition_data`, BIP как вложенный dict → нижний треугольник BIC). **Единицы METRIC как есть** (по решению автора): Pc=бар, Tc=K, MW=г/моль, без старых `MW/1000`/`Pc*10`. Ключ EOS (**MPR/SRK**) выбирается в окне экспорта (Брусиловский не имеет прямого E300-эквивалента). Сервис `gui/services/export_service.py` (`FORMATS`, расширяемо). Окно: выпадающий формат + EOS + файл-диалог сохранения.
- **Импорт E300** (кнопка **Import E300** на Projects, бывшая «soon»): `calc_core/Utils/E300Import.py::parse_e300` (устойчив к комментариям `--`/пробелам) + `project_service.import_e300`. **Свойства дека переносятся как есть** (MW/Tc/Pc/Vc/ω/shift/BIP): `evaluate_composition_data` заполняет структуру → значения дека перезаписывают её (`edit_component_properties` при Tc/Pc/ω пересчитывает EOS-коэффициенты — модель согласована). Импортируется на **живом EOS Брусиловского** (ключ дека MPR/SRK — лишь метка Eclipse). Модель создаётся и открывается сразу (в отличие от Excel — у дека уже есть все свойства, форма их бы пересчитала). Round-trip export→import проверен (Pc/Tc/MW сохраняются).
- **Settings** (меню **Settings → Constants & convergence…**): `gui/services/settings_service.py` — панель показывает **актуальные значения движка**: константа R, стандартные условия (P/T из `StandardConditions`), 6 критериев сходимости (`Constants.TOL_*`). Правки сохраняются в `gui_settings.json` (в `.gitignore`), но **пока НЕ влияют на расчёт** (движок читает свои модульные значения; проводка — отдельная задача, по решению автора). Save/Reset to defaults/Cancel, оранжевая заметка про «не применяется».

- **Заметка по надёжности**: DLE и Separator быстрые (~0.3с) и надёжные; **CCE тяжелее (~7с) и её Psat капризнее** (внутренние ретраи `SaturationPressure`/`sp_process None`) — доступна, но чинить сходимость движка вне scope (см. BACKLOG). В CI гоняется только DLE.
- **Идея (не реализовано)**: одновременный просмотр/сравнение двух результатов экспериментов (как Compare для флэшей) — распространить механизм Compare на эксперименты.

**Следующие правки/идеи (кандидаты, не начаты):**
- **Saturation pressure** (узел Psat) и **фазовая огибающая P-T** + критическая точка (implot уже подключён);
- визуальная схема-граф модели на imnodes (пост-MVP, «Фаза 4»);
- сравнение экспериментов между собой (сейчас Compare — только для флэшей);
- экспорт E300; импорт/экспорт Lab Data и сравнение нескольких прогонов между собой.

## Два главных драйвера дизайна

### 1. Реактивный вычислительный граф
- **Корневой узел** = состав + свойства (полностью редактируемый — см. «Редактирование состава»).
- От корня — узлы **экспериментов** (флэш, CCE, DLE, сепарация, огибающая, критическая точка, Psat), рядом с каждым — **сравнение с лабораторными данными**.
- **Реактивность**: правка корня → зависимые узлы помечаются «устаревшими». Стратегия пересчёта — **ленивая инвалидация** (пометить устаревшим) + кнопка **«пересчитать всё»**, а не авто-пересчёт всего на каждый чих (огибающая считается десятки секунд). Обязательны **индикатор прогресса и отмена**.
- Граф хорошо ложится на сервисный слой: каждый метод фасада (`flash`/`cce`/`dle`/`ssm`/…) = вычисление одного узла (вход = параметры, выход = результат, уже логируется в `result_store_object`).

### 2. Полностью настраиваемое рабочее пространство
- Главная претензия к PVTSim: **нельзя настроить под себя** — много лишних кнопок/полей/графиков, нет гибкости («как из окошка подводной лодки»).
- Хочется: панели/вкладки, которые можно **открыть, схлопнуть, переставить**. Вкладку модели открыл, когда нужна; не нужна — свернул. В рабочей области всё **динамично и настраиваемо**.
- **Раскладка панелей — часть состояния сессии**: при возврате к модели восстанавливается не только числовой результат, но и раскладка/открытые графики (продолжить с того же места).

## Стиль и выбор фреймворка (✅ решено 2026-07-16)

Выбран **DearPyGui** (нативный Python immediate-mode GUI поверх Dear ImGui + implot/imnodes). В запасе — **PySide**; крайний вариант, если понадобится доступ коллегам «по ссылке», — пересадка на **веб** (FastAPI + React/Plotly). Обоснование: встроенный node-editor (под вычислительный граф) + докинг панелей (под настраиваемое пространство), чистый Python поверх `calc_core` без API/JS, эстетика нравится автору. Известный минус — десктоп, не браузер; для localhost-v1 некритично.

**Тема оформления:** глобальная **светлая** тема (`gui/view/theme.py::build_light_theme`, палитра по мотивам ImGui `StyleColorsLight`, привязывается через `dpg.bind_theme` в `run()`) — по предпочтению автора.

**Ключевое следствие запасных вариантов — архитектура в два слоя:**
- **Фреймворк-независимый слой** (`gui/app_state.py` + `gui/services/`) знает только `calc_core`: состояние графа, статусы узлов, репозиторий моделей, запуск расчётов. **Не импортирует DearPyGui.**
- **Тонкий View** (`gui/view/`) — единственное место, знающее про DPG: рисует состояние, шлёт команды. Смена стека (PySide/веб) переписывает только `gui/view/`.

Оба запасных стека ложатся на тот же слой состояния/сервисов.

## План работ v1 (фазы)

Пакет GUI — top-level `gui/` (рядом с `calc_core/`, движок не трогаем). Запуск: `python -m gui` (зависимость `dearpygui`, опциональная группа `[gui]` в `pyproject.toml`).

- **Фаза 0 — Каркас** *(готово, ветка `gui/phase-0-1-dearpygui`)*: пакет `gui/`, точка входа `python -m gui`, главное окно с **закреплённой раскладкой** (слева прибитая панель дерева, справа рабочая область, снизу статус — без плавающего докинга), `AppState` (граф Проект→Модель→Вариант→Узел + статусы), файл сессии `gui_session.json`. Весь видимый текст — на английском.
- **Фаза 1 — Дерево + загрузка моделей** *(готово)*: левое дерево, список моделей из `models.json` через `Composition.from_db` (репозиторий `gui/services/model_repository.py`), открытие модели → таблица состава (корневой узел «Состав»).
- **Фаза 2 — Корневой узел (состав)** *(готово)*: рабочая область разбита на **вкладки** (tab-bar):
  - **Properties & correlations**: выбор EOS (`EOSType`) и корреляций C7+ (`gui/services/composition_service.py` — списки методов + обёртки), редактируемая таблица состава (inline-поля zi и свойств, правка через `edit_component_properties`), нормировка zi, кнопка «Recalculate properties» → `evaluate_composition_data`.
  - **BIP matrix**: полная симметричная матрица NxN, каждая ячейка — редактируемое поле; правка (i,j) зеркалится в (j,i) через `edit_bip_for_components`, диагональ фиксирована 0. Рендер матрицы 40×40 ~15 мс.
  - Смена корреляции помечает узел `STALE`, пересчёт → `FRESH`. Immediate-правки (zi/свойство/BIP) не дёргают перерисовку (без потери фокуса).
  - **Обе вкладки живут в отдельном окне** «Composition — <модель>», открываемом двойным кликом по модели в дереве (см. «Навигация» в Информационной архитектуре). Центральная рабочая область — под эксперименты.
- **Фаза 3 — Узел флэша** *(готово)*: в центральной области — ввод P (бар)/T (°C), кнопка «Run flash». Расчёт через `gui/services/flash_service.py` → `VLE.Flash` (обрабатывает и одно-, и двухфазный случай) в **воркер-потоке**; UI не висит, показывается `loading_indicator` + «Cancel». Результат применяется в главном потоке через `dpg.set_frame_callback`-опрос (DPG-safe). Панель результата из `FlashResult`: таблица Vapor/Liquid (доля фазы, M, молярный объём/плотность, density, Z, вязкость). **Реактивность (задел Фазы 4)**: правка состава/EOS/пересчёт помечают посчитанный флэш `STALE`. Отмена: движок непрерываемый, поэтому результат отбрасывается по завершении потока (UI остаётся отзывчивым). Узел `flash` заводится при открытии модели, `upstream=["composition"]`.
- **Фаза 4 — Реактивность + схема-граф**: ленивая инвалидация (правка корня → узлы `STALE`) + «пересчитать всё»; небольшая схема-граф модели (imnodes) — визуальная часть гибрида.
- **Фаза 5 — Сохранение сессии** *(готово)*: `gui_session.json` хранит последнюю модель, размер окна, было ли открыто окно редактора состава, параметры флэша (P/T), **сериализуемый слепок результата флэша** (`flash_service.snapshot_flash_result`/`restore_flash_result` — только отображаемые числа, без объекта состава) и время сохранения (`saved_at`). При старте, если есть восстановимая сессия, показывается **модальный вопрос** «A previous session was saved on \<дата\>. Continue?» (Continue → `_restore_session`: открыть модель, восстановить параметры+результат флэша, переоткрыть окно редактора; Start fresh → тихий старт, файл сессии сохраняется). Файл сессии — в `.gitignore`.

## Информационная архитектура

- **Слева — дерево**: `Проект → Модель → Варианты`.
  - **Вариант** = форк модели с другими свойствами / корреляциями / EOS / (в будущем) лампингом.
- **Навигация — дерево в стиле IDE (VS Code/PyCharm), реализовано 2026-07-16**: слева раскрывающееся дерево `Project → Model → (Composition / Flash / Saturation… )`; все модели выбранного проекта показаны соседними корневыми узлами. Клик по модели делает её активной и разворачивает; под ней — узел **Composition** и категории расчётов с историей запусков. Корневой контекстный пункт **Duplicate model...** создаёт новый соседний узел для адаптации варианта, а **Save model** сохраняет выбранную модель. Активный корень отмечен ASCII-маркером `[A]`, активный узел подсвечен в дереве. Клик по категории внутри неактивной модели также переключает контекст на неё. Состояние разворота хранится во View (`_expanded_models`/`_expanded_cats`).
- **Рабочая область — мультивкладки (как редакторы файлов в IDE)**: клик по узлу открывает/фокусирует его вкладку, крестик «x Close» закрывает (узел остаётся в истории). Composition — вкладка с внутренними под-вкладками Properties/BIP; каждый флэш — своя вкладка со своими P/T и результатом. Над вкладками — **хлебные крошки** (`Model > Flash > 50 bar/20 C · 2-phase`). Открытые вкладки/активная/история флэшей хранятся на `Variant` (каждая модель помнит свои вкладки) и сохраняются в сессию.
- Отдельное плавающее окно редактора состава (промежуточный вариант) убрано — Composition теперь вкладка.
- **Возврат к модели** → восстановление вкладок, истории флэшей (с результатами) и активной вкладки из сессии (после подтверждения в модалке).
- Реализационная заметка: таб-бар/вкладки/поля матрицы BIP и полей флэша создаются с **захватом id** (не фиксированные строковые теги) — иначе при пересоздании в том же кадре DPG падал с «Alias already exists» (освобождение алиасов отложено до следующего кадра).
