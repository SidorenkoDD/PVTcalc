# План развития PVTcalc

Актуальный порядок работ на **2026-07-24**. Структура документа намеренно
разделяет три состояния:

1. **Актуально** — то, что можно брать в работу сейчас;
2. **На паузе / заморожено** — задачи, которые не следует начинать без нового
   решения или внешних данных;
3. **Сделано** — единый нижний архив завершённых итераций.

Подробный фактический срез, результаты проверок и граница пользовательских
изменений находятся в [`AUDIT_2026-07-24.md`](AUDIT_2026-07-24.md).

## Актуально сейчас

### A1. Зафиксировать официальный PVTSim baseline — R0.3

**Приоритет P0 · CORE + QA · масштаб L · блокер внешних данных**

Официальный состав — **KRSNLN**. Нужны чистые входные и выходные данные
PVTSim, исходные единицы, версия и настройки симулятора, а также ожидаемые
Flash, Psat, DLE, CCE и Separator.

Excel-отчёты, найденные в рабочем дереве, пока не считаются baseline: они не
образуют версионированную пару input/output с автоматическими known-answer
assertions. Не реконструировать эталон из `models.json` и не объявлять экспорт
внешней проверкой.

**Готово, когда:** данные лежат в `tests/fixtures/`, тесты запускаются в CI,
а при расхождении видно физическую величину, абсолютную/относительную ошибку
и настройки расчёта.

### A2. Вернуть Cancel для Phase Envelope без потери скорости

**Приоритет P0 · CORE + GUI + QA · исследование M–L**

Сейчас Cancel работает для Flash, DLE, CCE и Separator, но отключён для SSM/Grid
в GUI. Thread-вариант ухудшил замер YAMB примерно с 4.47 до 13.0 секунд.

Нужны process-safe cancellation signal и замеры минимум на YAMB и KRSNL. Не
менять численные алгоритмы и допуски ради этой задачи и не возвращать
threading-fallback, пока не подтверждена производительность.

**Готово, когда:** остановка освобождает координатор, не применяет частичный
результат и не даёт существенной регрессии времени.

### A3. Перевести feature-view на объектную композицию — R2.5

**Приоритет P2 · GUI · масштаб M · следующий самостоятельный GUI-срез**

По одному заменить неявный mixin/MRO-контракт на объекты экранов с
`ViewContext` и небольшим `ViewHost`. Начать с изолированных
`dialogs_view.py` и `flash_view.py`, сохраняя headless smoke-тесты.

**Готово, когда:** `PVTcalcApp` остаётся composition root без длинного списка
`NotImplementedError`-заглушек, а feature-view тестируются отдельно.

### A4. Закрыть сквозные пробелы экспериментов

**Приоритет P1 · CORE + QA · масштаб M**

Добавить прямые regression-тесты для DLE, CCE и Separator поверх уже
протестированного Flash. Зафиксировать материальный баланс, условия ступеней,
поведение однофазных ветвей и явную ошибку несходимости. После этого можно
возвращаться к унификации Psat и CCE.

### A5. Разобрать полный Ruff по `calc_core`

**Приоритет P1/P2 · CORE + QA · серия малых изменений**

CI сейчас проверяет `ruff gui tests`; этот gate зелёный. Полная команда
`ruff check calc_core gui tests` на срезе ревизии сообщает **66 замечаний**.

Разбирать их отдельными сериями: сначала импорты и очевидная мёртвая логика,
затем живые модули. Не смешивать механическую чистку с изменением численных
алгоритмов и после каждой серии запускать regression-набор.

### A6. Очередь расчётов и «пересчитать устаревшие» — R3.5

**Приоритет P2 · GUI + CORE · масштаб M–L**

После process-safe решения A2 расширить координатор с одной worker-задачи до
явной очереди, добавить предварительный список и оценку объёма для
`Recalculate stale`. Каждый результат адресовать через полный `NodeRef`.

### A7. Упаковка и релизный процесс — R4.6

**Приоритет P1 · CROSS + QA · масштаб L · после стабилизации текущего пути**

Выбрать способ Windows-сборки, включить `DB.json`, тему и runtime DLL,
определить расположение пользовательских проектов/сессии/логов вне каталога
установки. Добавить версию, release notes, smoke на чистой машине и безопасную
миграцию схемы данных.

## На паузе / заморожено

### P1.2. Исправление критической точки — R1.2

Отложено решением автора. Текущий `CriticalPointCalculator` недостоверен для
многокомпонентных составов; GUI его не показывает. Возвращаться после R0.3 и
отдельного воспроизводимого failing-case с физически корректным критерием.

### P1.3. Единый Psat — R1.3

Отложено вместе с R1.2. Требуется единый `SaturationPoint` и решение, как
согласовать канонический путь с DLE/CCE/Separator, которые пока используют
`new_methodv2.SaturationPressure`.

### P1.4. Стабилизация CCE — R1.4

Отложено до R1.3. Нужно отделить проблемы Psat от расчёта ступеней,
ограничить retry-политику и добавить воспроизводимый CCE baseline.

### P1.5. Отдельный узел Saturation pressure — R3.1

Заблокирован R1.3: GUI-узел не вводить до согласования единого Psat-контракта.

### P1.6. Файловый импорт Lab Data — R3.3

Не планируется решением автора от 2026-07-22. Lab Data вводятся вручную,
вставкой из Excel или через существующий редактор проекта/модели.

### P1.7. Визуальный граф вычислений — R3.4

Не планируется решением автора от 2026-07-22. Пользовательская ценность не
подтверждена; не выполнять вместо улучшения рабочего UI.

### P1.8. Предметные расширения — R4.1–R4.4

Заморожены до отдельного запроса автора:

- CVD;
- лампинг/делампинг компонентов;
- EOS-regression;
- mixing rules и незавершённые C7+ корреляции.

Их не анализировать и не менять попутно с текущими задачами.

## Следующий релизный порядок

```text
A1 PVTSim fixture ───────────────┐
A2 process-safe Cancel ──────────┼─→ A4 experiment baselines
A3 feature-view composition ─────┘       ↓
A5 full calc_core lint ─────────────→ A6 calculation queue
                                      ↓
                                  A7 Windows release
```

Рабочая вертикаль уже существует:

```text
python -m gui
    → AppState / CalculationCoordinator / SessionState
    → gui.services
    → CompositionalModel / Flash / ExperimentsFacade / PhaseEnvelopeFacade
    → Composition → Stability → EOS → VLE → properties
```

Текущий проверенный срез: **305 тестов**, `ruff gui tests` и `mypy gui` проходят.
Полный Ruff ядра пока не является CI-gate.

## Сделано

Ниже собраны все завершённые пункты текущего roadmap. Исторические детали
сохраняются кратко; подробности реализации находятся в `BACKLOG.md`, `GUI.md`
и специализированных документах.

### R0 — воспроизводимая база

- **R0.1 ✅** — тесты отвязаны от пользовательского `models.json`; добавлены
  `tests/fixtures/models.json` и временное тестовое хранилище.
- **R0.2 ✅** — добавлены аналитические эталоны нормировки, EOS, Rachford–Rice,
  фазовых составов, материального баланса и однофазной идентификации.
- **R0.4 ✅** — создан пользовательский README с установкой, GUI, Python API,
  хранением данных, архитектурой и командами проверок.

### R1 — доверенное численное ядро

- **R1.1 ✅** — каноническая фазовая огибающая проверена: SSM — основной путь,
  Newton — независимая сверка, Grid — карта стабильности; введены диагностики
  причин NaN и физическая проверка bubble/dew.
- **R1.5 ✅** — закрыты edge cases PhaseEquilibriumNewton и покрыты 32
  зарегистрированные корреляции C7+.
- **R1.6 ✅** — добавлены `ResultDiagnostics` и `ResultWarning`, проверки
  физичности Flash и сохранение предупреждений в GUI.

### R2 — единый API и управляемые расчёты

- **R2.1 ✅** — `CompositionalModel` стал общей публичной точкой входа для
  Flash, экспериментов и фазовой огибающей; Flash возвращает `FlashResult`.
- **R2.2 ✅** — добавлен immutable `EngineConfig`, передаваемый в основные
  солверы и сохраняемый в снимке результата.
- **R2.3 ✅** — добавлены `CancellationToken`, progress callback и безопасное
  неприменение частичных результатов для Flash/экспериментов. Отдельная
  проблема SSM/Grid вынесена в актуальную задачу A2.
- **R2.4 ✅** — workspace v1, результаты в общей базе, provenance состава,
  атомарное сохранение и Save/Discard/Cancel lifecycle.

### R3 — инженерный GUI

- **R3.2 ✅** — Compare для однотипных Flash/DLE/CCE/Separator, Lab Data,
  абсолютных/относительных отклонений и полных `NodeRef`.

- **R3.7 ✅** — клавиатурная навигация, безопасные shortcuts, сообщения с

### Общие завершённые решения

- выбран канонический набор реализаций `calc_core/PhaseEnvelope/`;
- введена иерархия `PVTCalcError`;
- чтение и запись `models.json` централизованы в `ModelStore` с lock,
  revision, atomic write и backup;
- CI запускает pytest на Python 3.11/3.12/3.13 через Xvfb;
- добавлены физические диагностики, валидация входов и защита от скрытой
  мутации состава в фасадах;
- E300 import/export и ручные Lab Data работают в GUI;
- архитектурная карта GUI ↔ `calc_core` обновлена и сверена 2026-07-24.

## Правила обновления roadmap

- Выполненную задачу переносить в раздел **Сделано**, указывая дату и краткий
  результат.
- Отложенную или замороженную задачу переносить в **На паузе / заморожено**;
  не оставлять её среди актуальных.
- В **Актуально сейчас** держать только ближайшие задачи с понятным входом,
  критерием готовности и зависимостями.
- После численной задачи обновлять одновременно roadmap, backlog и тестовый
  baseline.
