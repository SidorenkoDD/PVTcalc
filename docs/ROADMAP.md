# План развития PVTcalc

Актуальный план развития проекта по состоянию на **2026-07-22**. Он объединяет
расчётное ядро, GUI и общепроектные задачи в одну очередь с понятным масштабом,
зависимостями и критериями готовности. История уже выполненных работ остаётся в
[`BACKLOG.md`](BACKLOG.md) и [`GUI.md`](GUI.md); этот файл — рабочая карта того,
что делать дальше.

## Как читать план

Область задачи:

- **CORE** — расчётное ядро `calc_core/`;
- **GUI** — состояние, сервисы и DearPyGui-представление `gui/`;
- **CROSS** — контракт между GUI и ядром, данные, документация, сборка;
- **QA** — тесты, эталоны и автоматические проверки.

Масштаб дан для одного разработчика и включает код, тесты и документацию:

| Масштаб | Оценка | Типичный характер работы |
|---|---:|---|
| **XS** | до 0,5 дня | локальная документация или механическая чистка |
| **S** | 1–2 дня | один модуль, простой экран или набор unit-тестов |
| **M** | 3–5 дней | несколько модулей и интеграционные тесты |
| **L** | 1–2 недели | новый сквозной сценарий или численный алгоритм |
| **XL** | больше 2 недель | исследовательская предметная функция с верификацией |

Приоритеты: **P0** — доверие к результатам или блокирующая предпосылка;
**P1** — ближайшее развитие продукта; **P2** — полезное продолжение;
**P3** — дальняя предметная перспектива. Оценка — ориентир, а не обещание срока:
для численных алгоритмов сначала обязателен короткий исследовательский этап.

Работа ведётся последовательно, начиная с первого незавершённого пункта. После
каждой законченной итерации её статус и фактический результат обновляются здесь
и в [`BACKLOG.md`](BACKLOG.md); незавершённый пункт не помечается выполненным.

## Текущая точка

У проекта уже есть рабочая вертикаль:

```text
python -m gui
    → AppState / CalculationCoordinator / SessionState
    → gui.services (валидация и адаптация данных)
    → CompositionalModel / Flash / ExperimentsFacade / PhaseEnvelopeFacade
    → Composition → Stability → EOS → VLE → свойства фаз
```

GUI умеет создавать, импортировать, редактировать, дублировать и сохранять
модели; считать Flash, CCE, DLE, Separator и фазовую огибающую; хранить
workspace модели, замеры и историю запусков; строить таблицы/графики и
экспортировать E300. Персистентность `models.json` защищена схемой, lock,
атомарной записью и резервными копиями.

Основные ограничения текущего состояния:

- тесты отделены от рабочей базы, но сравнение с PVTSim ещё не закреплено
  воспроизводимыми known-answer данными KRSNLN;
- критическая точка для многокомпонентных составов недостоверна, поэтому GUI
  намеренно её не показывает;
- DLE/CCE/Separator всё ещё используют `new_methodv2.SaturationPressure`, а GUI
  огибающей — канонический SSM; единый контракт Psat пока отсутствует;
- CCE медленнее и менее устойчива, чем DLE/Separator;
- отмена кооперативная: длинные циклы останавливаются в безопасных точках,
  отдельные старые атомарные поиски Psat могут завершить текущий шаг;
- настройки движка пока только читаются: immutable `EngineConfig` уже есть для
  расчётного API, но пользовательское редактирование ещё не подключено;
- `gui_session.json` больше не является источником результатов; legacy-снимки
  мигрируются при первом открытии модели, а Projects показывает только данные
  из сохранённого workspace `models.json`;
- CVD, лампинг/делампинг и регрессия EOS ещё не входят в живой продуктовый путь.

## Очередь релизов

```text
R0 — воспроизводимая база
  ↓
R1 — доверенное численное ядро
  ↓
R2 — единый API и управляемые расчёты
  ↓
R3 — законченный инженерный GUI
  ↓
R4 — новые PVT-возможности и выпуск приложения
```

Параллельно с R0–R2 можно делать только GUI-задачи, не зависящие от новых
численных результатов: сравнение существующих прогонов, импорт/экспорт замеров,
полировку навигации и документацию.
